### Gap Priority Ranking (SMB Developer Impact)

**P0 — Blocks revenue (must close before Jul 1 launch):**
1. **Published app hosting** — Lovable Cloud equivalent.  Users cannot share deployed apps without
   running `wrangler deploy` manually.  This is the #1 friction point for SMB users who want to
   show clients/stakeholders a live URL.  Proposed: Cloudflare Pages auto-deploy from CodeGeneratorAgent
   on `DEPLOY` command; zero-config, custom subdomain on `*.vibesdk.app`.
2. **Database auto-provisioning** — Lovable connects Supabase/Neon in one click.  vibesdk forces
   users to configure D1 themselves.  Proposed: auto-create D1 database per session with schema migration
   run on first `DEPLOY`; surface connection string in CodeGenState.
3. **Team collaboration** — Shared sessions would unlock team/agency use cases.  Lovable at GA.
   vibesdk architecture (per-user DO) makes this non-trivial but possible with session sharing RPC.

**P1 — Differentiator loss if not closed by Q4:**
4. **Mobile preview** — Simple iframe resize + CSS media query toggle.  1 sprint.
5. **Auth templates** — Clerk + Auth.js + Supabase Auth presets in SAAS_PAYMENTS_INSTRUCTIONS.
6. **Custom domains** — Cloudflare Workers custom domain binding via API; integrate into DEPLOY flow.

**Won't close (by design — architecture moat):**
- Design Canvas: Lovable's visual direct-manipulation layer requires a browser-in-browser rendering
  model.  Not viable in CF Worker environment.  Counter: our agentic engineering approach
  (DESIGN.md injection) produces better structural output than pixel manipulation.

### Decisions

**DEC-031-A:** Published app hosting (vibesdk Cloud) is **SPRINT 2 P0**.
Proposed implementation: `DEPLOY` WebSocket command → CodeGeneratorAgent calls CF Pages API
(via CF API token in Env) → returns `https://<session-id>.vibesdk.app` URL → surface in UI as
`DeployBadge` next to `IsolationBadge`.  No new infrastructure (CF Pages is Cloudflare-native).
Estimated effort: 3 days.

**DEC-031-B:** D1 auto-provisioning is **SPRINT 3 P0**.
Too risky for Sprint 2 (D1 API provisioning requires CF account token with write permissions;
needs careful per-user scoping).  Interim: inject D1 setup instructions into SAAS_PAYMENTS
system prompt so generated code includes correct D1 wrangler config.

**DEC-031-C:** Team collaboration is **SPRINT 4** — DO fan-out review needed first (ADR-001 §4).

## 3. NxCode Competitive Counter-Narrative

### Threat Profile (from run030 DEC-030-E)

NxCode: "free forever" model, targeting individual developers (not SMB teams).
Risk: price-sensitive users choose NxCode over vibesdk's paid tiers.

### Counter-Narrative Strategy

NxCode's "free forever" is viable only if:
- Single-user, no collaboration
- No quality guarantees (no eval gate)
- No security isolation (shared infra)
- No agentic engineering (single-pass generation)

vibesdk's differentiation is NOT price — it's **architecture quality**.

**Messaging framework:**

> "NxCode builds you an app. vibesdk builds you an *architecture*."
>
> Key proof points:
> 1. **Per-session isolation** — each build runs in its own Cloudflare Durable Object.
>    Your code never shares infrastructure with another user's code.  (IsolationBadge + SECURITY.md)
> 2. **Eval gate** — every generated phase is scored on faithfulness, relevancy, hallucination risk.
>    Bad output is blocked before it reaches your editor.  (EvalGate + PhaseQualityBadge)
> 3. **Parallel multi-agent** — Planner, Coder, Tester, Critic work simultaneously on independent
>    phases.  3× faster than single-agent generation on large apps.  (ADR-007 Option A)
> 4. **DESIGN.md** — inject your brand tokens into generation.  Output matches your design system,
>    not a generic template.

**Where to surface:**
- Landing page: "Why vibesdk vs free tools?" section (new component)
- Pricing page: comparison table row "Agentic engineering" vs "Single-pass AI"
- README: competitive positioning paragraph (after What-is section)

**Decision DEC-031-E:** Add "Why vibesdk vs free tools?" section to landing page — **SPRINT 2 P1**.
Minimal implementation: static markdown rendered in landing route.  Content: 4 proof points above.
No backend changes needed.

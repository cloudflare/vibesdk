## 1. Mastra v1.28–1.31 Backfill (April 22-May 1, 2026)

Previously captured: v1.32 (May 4) + v1.33 (May 13). New data on preceding releases:

**Source:** [GitHub releases](https://github.com/mastra-ai/mastra/releases)

### v1.30.0 — Durable Agents + Resumable Streams (April 30)

**What:** Durable agents with "resumable streams" for disconnect resilience. Workflow-backed durable execution via evented engine or Inngest.

**Relevance to vibesdk (HIGH):**
- vibesdk's `SimpleCodeGeneratorAgent` Durable Object already provides durability (SQLite-backed state, single-threaded per instance)
- Mastra v1.30 formalizes the pattern: long-running agent tasks survive network disconnects and can resume mid-stream
- **Specific gap:** vibesdk's `PhaseWorkflow.runPhaseWorkflow()` does NOT survive DO hibernation mid-phase. If the Mastra workflow step is interrupted (CF DO hibernation, 30s timeout), the phase restarts from scratch
- **Fix path:** Mastra v1.30 Inngest-backed durable execution = CF-compatible checkpointing (Inngest supports CF Workers via HTTP events). Wire PhaseWorkflow steps to Inngest for mid-phase resume on S11

**Effort estimate:** 2d (Inngest integration, step checkpointing, test coverage)

### v1.31.0 — Platform Channels + Slack Integration (May 1)

**What:** `ChannelProvider` framework + `@mastra/slack` with OAuth + threaded conversations.

**Relevance to vibesdk (MEDIUM):**
- Slack phase-complete notifications: when a phase finishes (PHASE_IMPLEMENTED), fire Slack webhook via ChannelProvider
- Implementation: register `SlackChannel` in Mastra config → broadcast to Slack thread on phase events
- Blocked by: Slack OAuth in CF Workers (fetch-based, CF compat confirmed)

**Action (S11 P2):** Wire ChannelProvider → Slack for phase completion alerts (adds "collab" dimension to vibesdk — team gets pinged when their app's latest phase ships).

### v1.28.0 — Modal Cloud Sandbox Provider (April 24)

**What:** `@mastra/modal` — Modal cloud sandbox integration for agent tool execution.

**Relevance to vibesdk (LOW):**
- vibesdk uses a custom container service for sandbox execution (already CF-native)
- Modal is a serverless GPU/CPU sandbox (Python-first, E2B competitor)
- vibesdk's sandbox is JavaScript/TypeScript native — Modal adds no value here

**Decision:** AVOID. Continue with custom container service.

### v1.29.0 — CF Workers MCP JSON Schema (April 28)

Confirms CfWorkerJsonSchemaValidator (already captured run017, shipped in SDK v1.23.0). No new action.

### v1.26.0 — Hardened Credential Leak Prevention (April 22)

Telemetry now scrubs credentials before logging. vibesdk uses `createLogger()` custom logger (not Mastra's default telemetry). No immediate action, but verify vibesdk logger doesn't log sensitive fields.

## 2. Claude Opus 4.7 — Production Impact (GA April 16, 2026)

**Sources:** [GitHub changelog](https://github.blog/changelog/2026-04-16-claude-opus-4-7-is-generally-available/), [Claude docs](https://platform.claude.com/docs/en/about-claude/models/whats-new-claude-4-7)

### New Tokenizer (1x–1.35x more tokens)
Opus 4.7 uses a new tokenizer. For the same input text, it may use **up to 35% more tokens** than Claude 3.x / 4.x models.

**vibesdk impact — CRITICAL for cost model:**
- `runEvalGate()` uses Gemini 2.5 Pro (not Claude Opus) — no impact
- `runPhaseWorkflow()` uses Claude claude-3-7-sonnet for plan + implement steps (see AGENT_CONFIG)
- vibesdk does NOT use Opus 4.7 today — impact is deferred until model upgrade
- **Action:** Before upgrading any operation to Opus 4.7, run token-count benchmarks on representative phase inputs. 35% cost increase on implementation steps = material per-session cost increase.

### 1M Context Window (No Premium)
Opus 4.7 exposes 1M context at standard pricing (previously required higher-cost long-context API).

**vibesdk opportunity:** The 1M context window makes whole-codebase context feasible for the eval step. Current Gemini 2.5 Pro eval uses a sampled file set (not full codebase). With Opus 4.7 at standard pricing, a full-context eval would be cost-competitive.

**Decision (S11):** Evaluate switching eval judge from Gemini 2.5 Pro to Claude Opus 4.7 (1M context, full-codebase eval). Gate on cost benchmark.

### High-Resolution Vision (2576px / 3.75MP)
First Claude model with high-res image support. vibesdk accepts image attachments at project creation (run via `imagesToBase64`). Current models cap at 1568px — Opus 4.7 unlocks richer UI mockup input.

**Action (S11 P2):** When upgrading blueprint generation to Opus 4.7, pass high-res mockup images for better blueprint fidelity.

## 5. Mastra v1.32/1.33 — Actionable Items (Backfill from prior research)

Items from run017 that need S10 action tracking:

### ResponseCache (v1.33) — on PhaseWorkflow
`ResponseCache` caches LLM step responses. For vibesdk's `PhaseWorkflow.runPhaseWorkflow()`:
- Cache hit rate is low for phase implementation (unique content each session)
- Cache hit rate is MEDIUM for plan step: if user re-runs the same prompt with same blueprint, plan could be cached
- **Action (S10 P2):** Enable `ResponseCache` on PhaseWorkflow plan step only. Implement with `sessionId` + `query` + `blueprintHash` as cache key.

### Agent Signals (v1.33) — Mid-run context injection
`agent.sendSignal()` allows injecting context into an active agent run.

**vibesdk opportunity:** User sends a message while a phase is running ("actually, make the background dark") → inject via `sendSignal` into the active phase implementation agent without restarting the phase.

Currently vibesdk queues user input in `pendingUserInputs` (state machine picks it up on next phase). Agent Signals would allow real-time steering within a phase.

**Action (S10/S11 P2):** Evaluate Agent Signals as replacement for `pendingUserInputs` queue in `PhasicCodingBehavior`.

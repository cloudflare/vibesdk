## 4. CF Agents SDK v0.12.4 — "Server Turn Persistence" Validates ADR-008 Pattern

**Released:** May 13, 2026  
**Source:** developers.cloudflare.com/changelog/post/2026-05-13-agents-sdk-v0124/

### Key feature: Server turn persistence
```
"keeps server turns running when a browser or client stream is interrupted"
cancelOnClientAbort: true parameter (optional — default is to keep running)
```

This is EXACTLY the WS buffering pattern ADR-008 S14 targets:
- Server-side inference continues regardless of client disconnect
- State buffered during disconnection
- Client reconnect retrieves buffered state

### Other v0.12.4 highlights
```
# schema: feature|relevance_to_vibesdk
Prevents duplicate initial state frames during connection setup|HIGH — vibesdk had this issue (deduplicated in frontend)
Improved recovery when tool calls span DO restarts|HIGH — same pattern as CodeGeneratorAgent WS recovery
Deferred user finish hooks until after agent startup|MEDIUM — similar to vibesdk's abort controller pattern
Structured tool output shapes preserved during message truncation|MEDIUM — relevant to PhaseWorkflow output
```

### ADR-008 S14 implementation signal
CF Agents SDK's `cancelOnClientAbort: true` pattern (keep-server-alive-on-disconnect) is the reference implementation for vibesdk's S14 app-layer WS buffering. vibesdk uses PartySocket + custom DOs (not CF Agents SDK), so the SDK feature doesn't auto-apply — but the pattern is validated and documented by CF's own implementation.

**S14 implementation plan (confirmed):**
1. In `CodeGeneratorAgent.ts` WebSocket handler: decouple inference lifecycle from client connection
2. When WS closes, do NOT abort the inference — continue buffering to DO storage
3. On reconnect (`agent_connected`), send all buffered messages from DO storage
4. Add `cancelOnDisconnect: boolean` option (default false = keep running)

## 5. CF Dynamic Workflows — Available, Evaluate S14/S15

**Released:** May 1, 2026  
**Source:** blog.cloudflare.com/dynamic-workflows/  
**Library:** `@cloudflare/dynamic-workflows` (MIT, ~300 LOC TypeScript)

### What it does
Extends CF Workflows durable execution engine so workflow code can differ per-tenant at runtime. Previously, CF Workflows required statically-deployed code. Dynamic Workflows adds a "Worker Loader" that routes execution to tenant-specific code when the engine wakes up.

### Relevance to vibesdk
vibesdk's code generation is already a state machine (IDLE → PHASE_GENERATING → PHASE_IMPLEMENTING → REVIEWING). Phase execution is stateful and resumable in theory, but currently:
- If the DO hibernates mid-phase, the phase must restart
- Parallel phase dispatch (ADR-007 Option A) doesn't have durable checkpointing

Dynamic Workflows could provide:
- Durable checkpointing per generation phase (survive DO restart mid-phase)
- Automatic retry with correct phase context
- Replace manual abort-controller + resume logic

**Assessment:**
```
# schema: aspect|verdict
Effort to integrate|HIGH (rewrite CodeGeneratorAgent as Workflow steps)
Value|HIGH (eliminate mid-generation DO restart failures)
Maturity|MEDIUM (released May 1, 2026 — very new)
vibesdk fit|MEDIUM (currently uses custom state machine, not Workflow model)
S14 scope|SPIKE DEFERRED — S15 evaluate after v0.12.4 WS fix ships
```

**Decision:** Do NOT block S14 on Dynamic Workflows. Implement ADR-008 app-layer WS buffering in S14 (faster, targeted). Evaluate Dynamic Workflows for S15 as a durable-execution upgrade path.

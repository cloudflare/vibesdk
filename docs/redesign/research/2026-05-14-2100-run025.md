## 1. Cloudflare Dynamic Workflows — Billing Starts May 26

run019 identified Dynamic Workflows (May 1, MIT + npm) as a fit for ADR-007 parallel phase dispatch. Billing detail now available:

**Billing model (starting 2026-05-26):**
- Bill: per unique Dynamic Worker created per day
- Unique = Worker ID + code — either change → new count, resets daily
- First 1,000 Dynamic Workers/day: free
- Over 1,000: $0.002/worker/day
- Requests + CPU time: standard Workers pricing (not extra charge)
- **Startup time billed separately from execution** (unlike standard Workers — parser + isolate init cost)

**vibesdk impact:**
Does vibesdk currently use Dynamic Workers? No — vibesdk uses Durable Objects (not Dynamic Workers). CodeGeneratorAgent instances are DOs, not Dynamic Workers. The `createDynamicWorkflowEntrypoint()` API would be used if we adopted CF Workflows for phase dispatch (was an ADR-007 option considered in run019 but not selected — ADR-007 Option A uses parallel DO fan-out instead).

**Decision: NO ACTION.** vibesdk's ADR-007 Option A (parallel DO fan-out via TeamLeadCoordinator) does not use Dynamic Workers. Billing starts May 26 but vibesdk is not affected. If we ever adopt Dynamic Workflows for phase orchestration (S13+), design Worker IDs for stability (same phase always = same ID = no daily re-billing).

## 2. DO Facets + Dynamic Workers — Per-Tenant App Architecture

**What shipped (blog.cloudflare.com, open beta):**
- Each Dynamic Worker can instantiate its own isolated Durable Object ("facet")
- Two SQLite DBs per DO instance: one for the supervisor (parent), one for the facet
- Zero-latency storage: facet DB lives on same machine as the DO
- Supervisor pattern: platform code (the parent DO) controls generated code as a Dynamic Worker facet
- Supervisor gets: resource limits, creation tracking, billing hooks, observability, logging

**Why this matters for vibesdk:**
vibesdk ALREADY implements this architecture manually:
- `SimpleCodeGeneratorAgent` (the supervisor DO) stores blueprint, files, git history in SQLite
- Each session is isolated; no shared state between sessions
- The pattern: supervisor DO (per-session) holds all state, user code runs in sandbox container

DO Facets would formalize this into CF's official pattern, enabling:
- Sub-DO per generated file set (extreme isolation beyond current per-session)
- CF-managed lifecycle for generated app instances
- Native resource limits + billing per generated app (vs current all-in-one session DO)

**Assessment:**
- Status: open beta, Workers Paid plan → vibesdk eligible
- Effort to migrate: HIGH (3 sprints) — fundamental DO topology change
- Current architecture already achieves the core guarantee (per-session isolation, SQLite-backed)
- **Decision: DEFER S13.** Document as validation of current approach. Counter-marketing: "vibesdk's DO-per-session architecture is the pattern Cloudflare now formally endorses as DO Facets."

## 3. CF AI Gateway — Unified Inference Layer (Agents Week)

**What changed from previous AI Gateway:**
- Before: routing proxy for CF-hosted models + third-party (requires HTTP client setup per provider)
- After: `AI.run()` Workers binding now works for **ALL** 70+ models across 12+ providers
- One-line provider switch: `AI.run('@anthropic/claude-sonnet-4-6', messages)` → same as calling CF-hosted model
- Auto-failover: if provider down, AI Gateway routes to backup automatically
- **Streaming resilience**: AI Gateway buffers streaming responses independently of agent lifetime → agent can disconnect + reconnect without re-invoking inference
- Unified billing: one wallet, one dashboard, per-agent cost tracking via custom metadata
- New providers: Alibaba Cloud, AssemblyAI, Bytedance, MiniMax, Pixverse, Replicate, Runway, Vidu
- Multimodal: image, video, speech models now in catalog

### Strategic impact for vibesdk

**Streaming resilience = DIRECT FIX for WebSocket reconnect issue (P1):**
Currently, if a user's WebSocket drops mid-generation, the agent must restart the LLM call from the beginning (or the call is lost). With AI Gateway streaming resilience:
- AI Gateway holds the LLM stream buffer
- Agent (DO) reconnects to AI Gateway mid-stream
- No LLM re-invocation needed
- Cost savings + better UX on mobile/flaky connections

This is a native platform feature, not something vibesdk needs to build. The work: ensure vibesdk's LLM calls route through `env.AI` binding (AI Gateway) rather than direct `fetch()` to provider. Currently, vibesdk uses direct fetch for Anthropic (claudeDirect.ts) and AI Gateway for Gemini/Grok. Moving Claude sub-agents to AI.run() gains streaming resilience + failover.

**vibesdk modelRouter → AI.run() migration path (S11 P2):**
```typescript
// Current (claudeDirect.ts):
fetch('https://api.anthropic.com/v1/messages', { headers: {...}, body: ... })

// Target (after AI.run() migration):
env.AI.run('@anthropic/claude-sonnet-4-6', { messages, max_tokens, system })
// Same result; gains: failover, streaming resilience, unified billing
```

Migration scope: `claudeDirect.ts` → thin wrapper over `env.AI.run()`. Keep ClaudeCallArgs shape unchanged; only swap the fetch internals. `ANTHROPIC_API_KEY` still needed in env until CF manages the key via Gateway credential.

**Estimated effort: 4h.** No API contract changes, no sub-agent code changes.

## 4. Mastra CloudflareDOStorage (March 2026)

Mastra released `CloudflareDOStorage` from `@mastra/cloudflare/do` (March 12, 2026):
- Extends `DurableObject`, init with `sql` (DO's SQLite) + optional `tablePrefix`
- Strong consistency: reads/writes serialized per DO
- Type-safe Mastra storage interface over raw DO SQLite

**vibesdk current:** PhaseWorkflow uses `AgentMemoryClient` (Mem0RestMemoryClient) for phase eval storage. The DO's raw SQLite is used directly for blueprint/files. Mastra's CloudflareDOStorage could replace direct `this.ctx.storage.sql` calls in SimpleCodeGeneratorAgent for Mastra-managed tables (plan_nodes, phase_eval).

**Assessment:** Low migration priority — raw DO SQLite already works and is faster (no Mastra abstraction layer overhead). Worth adopting for `plan_nodes` table only if Mastra's schema management becomes a maintenance burden. **DEFER S12.**

## 5. Competitor Infrastructure Signals (since run019)

### Replit Agent 4 — Parallel Sub-Agents Architecture Clarity
Agent 4 parallel execution confirmed: explore + test + build as simultaneous specialized sub-agents. Merge conflict resolution 90% automated via LLM-based merge. This is **per-container** (Replit runs containers, not DOs).

**vibesdk contrast:** DO-per-sub-agent (Planner/Coder/Tester/Critic) is more isolated than Replit's per-container approach. DOs provide single-threaded serialization + zero cross-agent shared state. Replit's 90% auto-resolve acknowledges the 10% failure rate — vibesdk's disjoint file assignment eliminates merge conflicts architecturally (100% guarantee, not 90% heuristic). This remains the strongest architectural differentiator.

### Cursor — Multi-Repo Cloud Agent Environments
Dockerfile + build secrets + audit logs + layer caching (70% faster). Architecture: Cursor agents each get a cloud VM with the full dev environment. Very different from vibesdk's DO-per-session + sandbox container. Cursor's model is heavier (full VMs) but supports arbitrary multi-repo scenarios.

**vibesdk response:** Single-app sessions (one repo, one project) are vibesdk's current scope. Multi-repo is a future feature. The DO + sandbox container approach is faster to spin up (< 1s) than a cloud VM. Feature gap is real for enterprise multi-repo workflows; not relevant for current target (non-dev founders).

### v0.dev (Vercel) — Sandbox Runtime + Git Panel
Sandbox-based runtime importing any GitHub repo + branch-per-chat + deploy-on-merge. Full git workflow for non-engineers. Architecture: Vercel sandbox (similar to E2B Firecracker) + Vercel deployment pipeline.

**vibesdk contrast:** vibesdk's isomorphic-git + SQLite filesystem adapter + Cloudflare Pages deploy is the equivalent stack. Already functional (git commit history, push to Pages). Gap: v0's Git panel is a UI surface that makes the workflow visible. vibesdk has the mechanics; the branch-per-chat UI is missing. **S11 P2: surface git branch creation as UI feature.**

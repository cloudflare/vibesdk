## 1. @cloudflare/think — API surface read

### Install
```bash
npm install @cloudflare/think agents ai @cloudflare/shell zod workers-ai-provider
```

### Execution Ladder (five tiers)
```typescript
// Tier 0: in-process workspace (shell commands)
createWorkspaceTools()

// Tier 1: Dynamic Worker (write + execute TypeScript code)
createExecuteTool()         // from @cloudflare/codemode

// Tier 2: npm support (Worker + npm resolution)
// via @cloudflare/worker-bundler

// Tier 3: Browser automation
createBrowserTools()        // Cloudflare Browser Run

// Tier 4: Full sandbox
createSandboxTools()        // Cloudflare Sandbox
createExtensionTools()      // Extension layer
```

**vibesdk touchpoint:** `worker/services/sandbox/factory.ts` currently implements a bespoke tier model. CF Project Think's Execution Ladder is a direct structural parallel — same conceptual tiers, different CF-native implementations. Lift-and-shift would replace bespoke factory with CF-maintained primitives.

### Fibers (durable execution)
```typescript
// Register a durable phase run
await runFiber("phase-gen", async (ctx) => {
    // … do work …
    ctx.stash({ generatedFiles });   // checkpoint state
});

// Recovery hook (called on crash/restart)
onFiberRecovered(ctx) {
    // ctx.name = "phase-gen", ctx.snapshot = { generatedFiles }
}
```

**vibesdk touchpoint:** `getOrCreateAbortController()` in `SimpleCodeGeneratorAgent` handles abort. Fibers add **crash recovery** on top — if the DO crashes mid-phase, `onFiberRecovered` restores from last stash. This is strictly additive to vibesdk's current pattern and closes the "what happens to a 40-min session if the DO crashes" question without bespoke logic.

### Sub-agents
```typescript
const researcher = await this.subAgent(ResearchAgent, "research");
const result = await researcher.chat(query, streamRelay);
// Each sub-agent gets its own SQLite database (typed RPC)
```

**vibesdk touchpoint:** Cycle 2/3 escalated parallel sub-agents as table-stakes. `this.subAgent(AgentClass, "name")` is the CF-native equivalent of DO-per-sub-agent. Isolation is enforced by the SDK. TypeScript compile-time safety on the RPC.

### Breaking changes (v0.12.4, May 13 2026)
- `ChatOptions.tools` **REMOVED** from TypeScript API
- Default message pruning behavior changed in `@cloudflare/think`
- `cancelOnClientAbort` ADDED to `useAgentChat()`
- `routingRetry` configuration ADDED to `getAgentByName()`

**vibesdk impact:** `ChatOptions.tools` removal is the most significant — any vibesdk code using that API shape would need migration. Current `buildTools()` and `buildDebugTools()` in `worker/agents/tools/customTools.ts` use the worker-level tool pattern, not `ChatOptions.tools` directly, so the risk is low but needs verification during spike.

### Maturity verdict

## 2. Vercel Fluid Compute vs CF Workers — updated picture

### CF's official benchmark response (blog.cloudflare.com)
- **After benchmark fixes:** "CF performs on par with Vercel in every case except Next.js"
- **What CF fixed:** 1 vCPU (not 2) for single-threaded workloads; AWS us-east-1 network (not SF laptop); Next.js force-dynamic configuration corrected; React NODE_ENV variable fixed
- **V8 GC tuning:** ~25% boost to benchmarks
- **JSON.parse() reviver:** ~33% speedup pushed upstream to V8
- **Next.js gap:** Still exists — CF does not claim parity on Next.js SSR

### Durable Objects — no published hot-path numbers
- Community benchmark: DO storage p99 ~22ms (read latency; storage layer, not full agent path)
- CF Queues blog: "DO median latency dropped from ~200ms to ~60ms" (for queues use case)
- No p50/p95/p99 for streaming token output from a DO agent available anywhere

**vibesdk opportunity:** vibesdk is the only party that can publish realistic DO agent hot-path timing (streaming token, phase-gen, eval gate). A 3-day instrumentation effort + 1 week of production data = a publishable benchmark that counters the Fluid Compute 2.55× narrative with real DO numbers (not SSR proxy numbers).

### Vercel Fluid Compute asymmetry — confirmed
- Workers: 128 MB shared CPU
- Fluid Compute: 2 vCPU + 4 GB RAM
- Guillermo Rauch (Vercel CEO) on X: "1.2× to 5×" for server rendering
- **This is not an apples-to-apples test.** CF's response is that with matched 1 vCPU, parity is achieved except Next.js.

**The narrative risk is real regardless:** Fluid Compute 2.55× claim will be cited by buyers without reading the methodology footnotes. vibesdk needs its own DO benchmark to break out of the "Workers is slow" framing.

## 3. Terminal Use (YC W26) — product launch analysis

### What shipped
```
# schema: feature|status
deployment-model|config.yaml + Dockerfile → CLI deploy (similar to Vercel CLI)
lifecycle-model|three endpoints: on_create, on_event, on_cancel
agent-sdk-adapters|Claude Agent SDK + Codex SDK (built-in)
frontend-compat|Vercel AI SDK v6
filesystem-model|persistent forkable FS, decoupled from task lifecycle
environments|preview/production, git-based deploy, logs, rollback
filesystem-sharing|multiple agents can share one FS
```

### Sandbox tech (UNCONFIRMED)
- Co-founder comment: "FUSE performance overhead is a constraint we're working around"
- Community theory: "copy-on-mount backed by object storage (S3/R2/GCS)"
- **Not** confirmed as Firecracker. No container technology named.

### Comparison to vibesdk architecture

| Dimension | Terminal Use | vibesdk |
|---|---|---|
| State persistence | Forkable FS (object-storage-backed) | SQLite-FS adapter inside DO |
| Agent isolation | Physical compute isolation per customer | One DO per session |
| Lifecycle model | `on_create/event/cancel` | Durable Object RPC methods |
| SDK support | Claude Agent SDK, Codex SDK adapters | Custom Anthropic + Gemini toolchain |
| Filesystem | First-class, shareable, forkable | Git-backed SQLite FS (`fs-adapter.ts`) |
| Deploy model | Vercel-style CLI deploy | CF Wrangler deploy |

**Key insight:** Terminal Use treats the FS as the primary primitive; agents are consumers of a FS. vibesdk treats the DO as the primary primitive; FS (SQLite) is an implementation detail inside the DO. These are complementary, not identical architectures.

**Threat level:** MEDIUM. Terminal Use makes it trivial to deploy _any_ agent (Claude SDK, Codex) with persistent FS + git deploy. If they add a code-gen workflow on top, they become a substrate that competes with vibesdk's entire agent runtime. For now: infra-layer threat, not product-layer threat. Watch: Firecracker confirmation, pricing, enterprise features.

**Opportunity:** Terminal Use's Claude Agent SDK adapter is a distribution path — vibesdk could expose a Terminal Use-compatible deployment mode for teams that want to self-host their code-gen agent. Low-effort, expands TAM.

## 2. DO Facets + Dynamic Workers — Architecture Constraint Identified

**Source:** CF Blog post "Durable Objects in Dynamic Workers: Give each AI-generated app its own database"; CF DO changelog

### What DO Facets actually require

DO Facets are NOT a standalone DO feature — they are **a feature of Dynamic Workers**.

```
# schema: requirement|detail
Dynamic Worker Loader API|REQUIRED — generated app code must be loaded via the Worker Loader API
Workers Paid plan|REQUIRED — Facets are in open beta, Paid plan only
wrangler.jsonc additions|dynamic_dispatch_worker binding needed
API surface|this.ctx.facets.get(name, () => ({ class: appClass })) — from inside DO only
SQLite isolation|Each facet has own SQLite; cannot read parent DO's DB
facet.fetch()|Forward HTTP requests into the facet's code
RPC methods|Available on facet handle same as regular DO RPC
```

### Current vibesdk generated-app deployment path
```
Current: React frontend + optional CF Worker backend → standard wrangler deploy
Required for Facets: Generated app code must go through Dynamic Worker Loader API
Gap: vibesdk generates static wrangler.toml deployable bundles, not dynamic-dispatch workers
```

### S14 spike revised scope

**Original S14 estimate:** 2d spike for DO Facets
**Revised:** DO Facets + Dynamic Worker Loader = 3-5d. Escalate to S15 unless Dec 15 target for owner forces earlier.

**Key implementation questions for spike:**
1. Can vibesdk use `ctx.facets.get()` inside `SimpleCodeGeneratorAgent` (already a DO) to provision per-session SQLite for generated apps — without making the generated app itself a Dynamic Worker?
2. If yes: generated app connects to session DB via DO RPC → Facet's SQLite. No Dynamic Worker needed on the app side.
3. If no: full Dynamic Worker deployment required, which changes generated app output format.

**Option A (simpler — within existing DO):**
```typescript
// In SimpleCodeGeneratorAgent (already a DO):
const appFacet = await this.ctx.facets.get(`app-${sessionId}`, () => ({
    class: AppSQLiteStore, // a simple DO class for storage
}));
const db = await appFacet.getDatabase(); // access the isolated SQLite
```
This avoids Dynamic Workers for the generated app. The generated app backend uses the facet's SQLite via DO RPC. **Lower effort, higher compatibility.**

**Option B (full Dynamic Workers):**
Generated app code is loaded via Worker Loader API at deploy time. Full isolation. Higher effort, requires generated app to be a Dynamic Worker, not a standard Cloudflare Worker bundle. Deferred to S15.

### Decision
Run S14 spike against Option A only. Time-box to 2d. If `ctx.facets.get()` supports the simpler DO-within-DO pattern without Dynamic Workers, ship. If not, defer to S15 as Option B.

### `cancelOnClientAbort` deep dive
```typescript
// CF Agents SDK pattern (v0.12.4):
const stream = await agent.run({ cancelOnClientAbort: true });
// When client disconnects: server-side inference is cancelled

// vibesdk equivalent (ADR-008 app-layer buffer):
// When getWebSockets().length === 0 → buffer messages, don't cancel inference
// This is the OPPOSITE behavior: vibesdk keeps running, SDK offers cancel option
// Both patterns are valid — different tradeoffs:
//   SDK cancelOnClientAbort: cancel = no waste on abandoned sessions
//   vibesdk buffer: continue = user can reconnect and see buffered progress
```

**Assessment:** vibesdk's ADR-008 "continue + buffer" approach is the correct choice for our use case — users frequently disconnect during long builds (5-10 min) and reconnect. Canceling on disconnect would waste the inference investment. The `cancelOnClientAbort: false` behavior is what we implement.

## Cloudflare Platform Catchup (Agents Week 2026)

| Feature | Type | vibesdk impact |
|---|---|---|
| Dynamic Workflows (MIT, open beta) | Infra | HIGH — see note |
| Durable Object Facets | Infra | Awareness only; validates existing DO-per-session design |
| CF Mesh | Networking | DEFER — private DB/API access via DO already achieved |
| CF Email Service (beta) | Communication | Opportunity for transactional email without Resend/SES |
| CF Sandboxes | Compute | Future: persistent shell + FS = could replace custom container |
| Agents SDK improvements | SDK | Chat recovery, Voice agent — monitor, no action needed |

**Dynamic Workflows — critical read:**

Dynamic Workflows is an MIT-licensed TypeScript library (~300 lines) enabling per-tenant/per-agent durable workflow logic at runtime. Previously, CF Workflows required fixed deployment-time code.

Architecture: Worker Loader sits between Workflows engine and tenant code. When `env.WORKFLOWS.create()` is called, the system wraps with tenant metadata, persists it, routes execution back to correct tenant code on wake.

Competitive claim: "Runtime code loading + V8 isolate-level isolation + edge distribution is currently unique to Cloudflare's stack" vs Temporal, Inngest, AWS Step Functions.

vibesdk angle: Our PhaseWorkflow.ts uses static Mastra workflow definition per deploy. Dynamic Workflows would allow PhaseWorkflow logic to differ per session/app-type at runtime — enabling, e.g., a payments-specific durable workflow vs a dashboard-specific one. This is a future-phase consideration (S13+), but ADR-001 note needed.

## Mastra Version Catchup (v1.29–1.33)

| Version | Date | Key Feature | vibesdk action |
|---|---|---|---|
| 1.29.0 | Apr 28 | Forked subagents; CF Workers V8-isolate MCP JSON schema | None needed |
| 1.30.0 | Apr 30 | **Durable Agents** (resumable streams); Workflow-backed durable execution | ADR-008 overlap — note |
| 1.31.0 | May 1 | Channels Framework; Slack integration; NestJS adapter | None needed |
| 1.32.0 | May 4 | **FGA via WorkOS**; Scheduled Workflows + cron; MCP Apps | Cron: consider replacing research cron |
| 1.33.0 | May 13 | **ResponseCache**; Agent Signals; PubSub; Azure OpenAI + WebSocket | ResponseCache = HIGH |

**Mastra v1.30 Durable Agents overlap with ADR-008:**
Mastra v1.30 ships resumable streams for crash/disconnect resilience — the SAME problem our ADR-008 (CF AI Gateway streaming buffer) addresses via CF infrastructure. Two approaches:
1. CF AI Gateway RFC (ADR-008): infra-level buffer, session replay via `X-AI-Gateway-Durable-Id`
2. Mastra v1.30 Durable Agents: application-level, workflow-backed resumption

These are complementary (CF handles transport; Mastra handles orchestration state), but the overlap confirms ADR-008 remains PROPOSED-deferred (S12) rather than urgent.

**Mastra v1.33 ResponseCache — confirmed viable path:**
`ResponseCache` processor caches identical LLM steps. Streaming paths incompatible (confirmed in iter 35). Non-streaming paths that ARE viable:
- `generate-blueprint.ts` → `generateBlueprint()` → planning LLM call
- `EvalGate` scorer calls inside `PhaseWorkflow.ts evalStep`

Priority: Implement ResponseCache on evalStep first (highest hit rate for retry paths).

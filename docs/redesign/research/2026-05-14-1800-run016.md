## What this means for vibesdk (features pillar, cycle 4 additions)

1. **MCP is now bilateral — consumer AND provider.** Lovable exposes itself as an MCP server (May 7). Bolt connects to external MCP servers. vibesdk has zero exposure. The MCP gap has crossed the line from "table-stakes delayed" to "competitive liability." A vibesdk MCP server that exposes `create_phase`, `run_phase`, `get_phase_status`, `get_eval_verdict` as MCP tools would let Claude Code users control vibesdk from the terminal — exactly the dev workflow that wins enterprise. Effort: 2 sprints.

2. **Context Usage Breakdown is a fast UX win.** Cursor's May 6 dashboard shows context consumption per category (rules, skills, MCPs, subagents). vibesdk broadcasts `STATE_DELTA` with eval scores; extending it to include session token counts (`systemPromptTokens`, `conversationTokens`, `phaseContextTokens`) in the badge or a debug drawer is 2-3 days of work. Closes an enterprise observability gap.

3. **Replit micro VM branching → market vibesdk git-DO branching.** "Branching feels instant using micro VMs" is Replit's marketing message for the same thing vibesdk already does via git-backed SQLite FS within a DO. Write the counter-marketing: "vibesdk branches are git tree operations — no VM cold start, no additional compute cost, full git history preserved."

4. **Publish-time security scan: ship or formally drop.** Lovable AND Replit both have it. Cycle 2 ROI #6 said "30-day window." That window is now closed. Either ship a minimal Semgrep-on-publish hook this sprint or remove from roadmap and accept the gap.

5. **Parallel sub-agent merge strategy.** Replit's 90% auto-resolution is a product signal. When vibesdk ships parallel sub-agents, phases modifying the same file will conflict. Design the merge strategy before implementation: (a) phase independence constraint (no two phases touch same file), (b) automated LLM-mediated merge (like Replit), or (c) human-in-the-loop merge UI.

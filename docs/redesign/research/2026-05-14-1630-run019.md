## 1. Cloudflare Dynamic Workflows (May 1, 2026)

**Source:** [CF Blog](https://blog.cloudflare.com/dynamic-workflows/) | [InfoQ](https://www.infoq.com/news/2026/05/cloudflare-dynamic-workflows/)

**What:** MIT-licensed npm library that routes durable Workflow execution to tenant-supplied code at runtime. Each tenant, agent, or request can provide a different `WorkflowEntrypoint` implementation — dispatched dynamically via Dynamic Workers.

**API surface:**
```typescript
// Platform code (vibesdk's TeamLeadAgent)
import { createDynamicWorkflowEntrypoint, wrapWorkflowBinding } from 'dynamic-workflows';

// Wrap existing workflow binding to add dynamic dispatch
const DynWorkflow = createDynamicWorkflowEntrypoint({ /* routing config */ });

// Dispatch per-agent workflow (parallel)
const run1 = env.WORKFLOW.create({ id: 'phase-auth', code: coderAgentCode });
const run2 = env.WORKFLOW.create({ id: 'phase-ui', code: coderAgentCode });
await Promise.all([run1, run2]);
```

**Relationship to vibesdk S10:**
- ADR-007 Option A (Phase Independence Constraint) = disjoint file sets + `Promise.all()` dispatch
- Dynamic Workflows provides the parallel dispatch infrastructure ADR-007 assumes
- Each CoderAgent phase could be a separate Dynamic Workflow instance with its own execution context
- Replaces the manual `Promise.all(phases.map(p => this.executePhaseImplementation(p)))` with durable execution

**Decision impact for ADR-007:**
- Option A remains correct primary choice (no merge = correct by construction)
- Option B (LLM merge) would naturally use Dynamic Workflows' branching model
- The `wrapWorkflowBinding()` pattern fits vibesdk's PhaseWorkflow exactly

**Availability:** Open beta, Workers Paid plan (Dynamic Workers required).

| Dimension | Included (Free tier) | Overage |

| Dynamic Workers/day | 1,000 unique/month | $0.002/worker/day |
| Requests | 10M/month | $0.30/M |
| CPU time | 30M ms/month | $0.02/M ms |

**A unique worker = `(workerId, code)` tuple.** If code changes, new billing unit.

**vibesdk cost projection (S10 parallel sub-agents):**
- 4 CoderAgents per session × $0.002 = $0.008/session overhead
- 1,000 sessions/day × $0.008 = $8/day = $240/month in agent overhead
- At Pro plan ($20/user, ~50% margin target): 1,000 sessions = ~500 active users → manageable
- **Action:** Use stable CoderAgent code hash per version to minimize unique worker count. Pin worker ID to `{version}-{agentType}` to reuse within billing day.

**Beta period:** Free until May 26. vibesdk should start S10 parallel sub-agent implementation NOW to complete before billing starts.

## 3. CF DO Facets (Open Beta, April 13, 2026)

**Source:** [CF Blog](https://blog.cloudflare.com/durable-object-facets-dynamic-workers/)

**What:** Dynamically loaded DO classes with isolated SQLite. A parent DO ("supervisor") loads tenant code and creates "facets" — each facet has its own SQLite database.

```typescript
// vibesdk's existing architecture IS DO Facets semantically:
// - SessionDO (supervisor) → spawns CoderAgent DOs (facets)
// - Each CoderAgent has its own SQLite (git FS + blueprint)
```

**Gap:** vibesdk manually manages this pattern via `getAgentStub()`. DO Facets formalizes it with:
- `this.ctx.facets.get()` — retrieve/initialize facet (vs. manual DO stub creation)
- Per-facet SQLite (already have this per-DO)
- Simpler lifecycle management

**Migration consideration:** vibesdk's current multi-DO architecture is functionally equivalent. Migration to DO Facets API would clean up code but is not blocking. DEFER to S11.

## 4. Mastra v1.32.0 — MCP Apps + FGA (May 4, 2026)

**Source:** [Mastra Releases](https://github.com/mastra-ai/mastra/releases)

### MCP Apps (v1.32.0) — HIGH relevance

New: `listResources()`/`readResource()` on MCPServer. Allows exposing UI resources over MCP — not just tools.

**For vibesdk MCP server (S9 shipped):**
- Current: 3 read-only tools
- Upgrade path: expose generated files as MCP resources (`vibesdk://session/{id}/files/{path}`)
- UI resources could render file diffs, preview URLs, quality dashboards in MCP clients

**Action (S10):** Add `registerResource()` calls to vibesdk MCP server for generated file access. This enables the `vibesdk_get_files` tool we deferred (file content as MCP resources is cleaner than JSON in tool output).

### Scheduled Workflows (v1.32.0)

`schedule` declaration in `createWorkflow()` → cron-native. Less relevant for vibesdk (phases are triggered by user input, not cron). But could be used for the research loop (replace CronCreate with Mastra scheduled workflow).

### FGA — Fine-Grained Authorization (v1.32.0)

`IFGAProvider` + `checkFGA()` + WorkOS-backed provider. Relationship-based authorization.

**For vibesdk MCP tools:** Current ownership check = manual DB query per tool.
FGA would replace this with a policy layer. Low priority — current pattern is simpler and correct.

## Architecture Implications for S10 (Parallel Sub-agents)

### Revised S10 Implementation Path (ADR-007 + New CF Primitives)

```
Current (S9 shipped):
  TeamLeadAgent → sequential executePhaseImplementation() × N phases

S10 target (ADR-007 Option A + Dynamic Workflows):
  TeamLeadAgent:
    1. validateDisjointFiles(phases) → conflict check
    2a. No conflict: dispatchWorkflow(coderCode, phaseData) × N → Promise.all()
    2b. Conflict: sequential fallback
    3. Aggregate results → composite EvalGate

Infrastructure:
  - Dynamic Workflows: durable parallel dispatch (vs. raw Promise.all)
  - Phase independence: ADR-007 Option A ensures no branch merging needed
  - DO Facets API: optional cleanup (DEFER S11)
```

## 5. Agent Signals — Potential Integration

### From run023/run027: Mastra Agent Signals (v1.33)

`subscribeToThread(threadId, handler)` — inject context into a running agent mid-execution.
Allows external events to influence an in-flight LLM call.

**vibesdk current:** `pendingUserInputs` queue in `CodeGenState` handles mid-generation
user interrupts. Agent Signals would be a more elegant API for this.

**Integration assessment:**
- Mastra `subscribeToThread` uses Mastra's own thread storage (not vibesdk's DO SQLite)
- Migration would require vibesdk's session threads to be registered with Mastra's storage layer
- Not compatible with current stateless `PhaseWorkflow` (workflow recreated per session)
- Estimated: S12 evaluation spike (1 day), S13 implementation if viable

**Decision DEC-032-E:** Agent Signals evaluation spike — DEFERRED to S12 after vibesdk Cloud
(DEC-031-A) ships. Rationale: pendingUserInputs queue is working; prioritize visible features
over architectural refactors in Q3 window.

## 6. Mastra ResponseCache for planStep (from Open Threads)

### Background

eval-cache.ts (iter 39) caches `runEvalGate` results by (sessionId, phaseName).
The open thread from run031 proposes similar caching for the blueprint generation step.

### Blueprint generation flow (from phasic.ts)

```
phasic.ts:runPhaseWorkflow()
  → PhaseWorkflow.run()
    → planStep: generateBlueprint() → Claude Sonnet 4.6 LLM call
    → implementStep: ... (per phase)
    → evalStep: runEvalGate() [CACHED via eval-cache.ts]
```

The `planStep` calls `generateBlueprint()` which makes a full LLM call. This is expensive
($0.03-$0.15 per blueprint depending on complexity) and happens on:
1. Session start (first blueprint generation)
2. Every `runPhaseWorkflow()` call (currently not cached — regenerates each time)
3. WebSocket reconnect if the session is re-initialized

**Cache design (follow eval-cache.ts pattern):**

```
Key: `blueprint:${sessionId}:${blueprintHash(userQuery + projectType)}`
TTL: 30 minutes (blueprint is stable for a session; user can force-refresh)
Max size: 50 entries (blueprints are larger than verdicts)
```

**Cache value:** The full `Blueprint` object (phase list + instructions)

**Why TTL 30min (vs 10min for eval-cache):** Blueprint is unlikely to change within
a session. If user modifies the query significantly, the hash changes → cache miss → fresh generation.

**Decision DEC-032-F:** Implement blueprint-cache.ts (analogue to eval-cache.ts) in S12.
Place in `worker/agents/operations/blueprint-cache.ts`. Key = hash of (sessionId, userQuery).
Wire into `phasic.ts:runPhaseWorkflow()` before the `planStep` call.
Estimated effort: 2h implementation + 8 unit tests (same pattern as eval-cache.ts).

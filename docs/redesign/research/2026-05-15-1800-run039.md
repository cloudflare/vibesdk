## 3. CF AI Gateway RFC — Unchanged, ADR-008 Still Blocked

RFC #1257 (cloudflare/agents): "AI Gateway as durable response buffer for long-running inference calls" — still RFC-stage as of April 2026. No GA announcement.

### New signal: Agents SDK v0.12.4 chat recovery (application-layer alternative)

CF Agents SDK v0.12.4 (May 2026) added:
- **Server turns continuing during client interruptions** — the DO continues processing the LLM call even when the WebSocket client disconnects
- **Enhanced DO routing with retry support** — routes requests to the correct DO instance with backoff
- **Durable Think submissions** — LLM reasoning calls survive client disconnection

This is the **application-layer** approach to the same problem ADR-008 addresses at the **infrastructure layer**. The AIChatAgent in CF Agents SDK now has the buffering behavior built-in (DO holds the stream, client reconnects and receives it).

### Architecture comparison

```
# schema: approach|where_implemented|status|vibesdk_path
CF AI Gateway buffer RFC|Infrastructure (gateway)|RFC-stage BLOCKED|Wait — may never ship
CF Agents SDK AIChatAgent|Application (DO+SDK)|GA in v0.12.4|Requires migrating to AIChatAgent pattern
vibesdk custom DO+WS|Application (custom)|Shipped but no reconnect buffering|ADR-008 Phase 1 option
```

### Revised ADR-008 assessment

The Gateway RFC may be indefinitely deferred — CF is solving the problem at the SDK/application layer instead. For vibesdk, the options are:
- **Option A (preferred S14):** Implement application-layer buffering in CodeGeneratorAgent DO — emit/buffer chunks in DO SQLite, replay on reconnect. This is what AIChatAgent does. Medium complexity, 1–2 days.
- **Option B (wait):** Wait for AI Gateway RFC to GA. Risk: may never ship; CF appears to have moved the solution to SDK layer.

Recommendation: Plan Option A for S14. ADR-008 status changes from "BLOCKED waiting for RFC" to "BLOCKED until S14 application-layer implementation planned."

## 4. DO Facets — MAJOR New CF Primitive (Open Beta, Workers Paid)

Released April 13, 2026 (open beta, Workers Paid plan).

### What it does

Durable Objects Facets allow code loaded via Dynamic Workers to instantiate and manage its own Durable Object, with its own SQLite database. The facet's database is separate from the supervisor DO's database but stored as part of the same overall DO.

```
Supervisor DO (vibesdk session)
  └── generates app code
  └── Facet: UserApp DO (facet, separate SQLite)
        └── app-specific state, real-time features
        └── zero idle cost
```

### Relevance to vibesdk

vibesdk currently generates D1 setup docs (D1ProvisionService, iter 53) for user app persistence. DO Facets offers a complementary pattern: the session DO can provision a facet DO for the generated app at generation time, not requiring the user to manually set up D1 via Cloudflare dashboard.

Use cases for generated apps:
- Real-time features (chat, collaboration, live counters) — DO is better than D1 for sub-millisecond state
- Session-specific state (game state, shopping cart, ephemeral data) — no cold start vs KV
- Multi-tenant generated SaaS — each generated app's users get their own facet

### Assessment

| Question | Answer |
|---|---|
| Ready for S13? | No — still beta; D1ProvisionService sufficient for S13 |
| Value for vibesdk? | HIGH — "give each generated app its own database" is the exact vibesdk pitch |
| When to evaluate? | S14 spike: prototype facet provisioning as part of D1ProvisionService evolution |
| Pricing risk? | Workers Paid plan required — vibesdk already on paid plan |

## 5. CF Agents SDK v0.12.4 — Unified Tracing Cross-Worker

New in May 2026 CF changelog:
- **Unified Tracing**: single trace across Worker-to-Worker subrequests, trace context propagates automatically
- Enables cross-DO debugging in multi-agent architectures

### Relevance to vibesdk

vibesdk's multi-agent DO fan-out (Planner/Coder/Tester/Critic) currently lacks unified trace context across DOs. Unified Tracing would allow a single trace to span:
- `TeamLeadCoordinator` (supervisor)
- `PhasePlanner` (child DO)
- `PhaseCoder[i]` (parallel child DOs)
- `PhaseEval` (eval gate DO)

**Assessment:** LOW urgency (S14 observability sprint). The existing OpenTelemetry spans in Mastra PhaseWorkflow cover the in-process tracing. Cross-DO tracing is a production debugging improvement, not a user-facing feature. Defer S14.

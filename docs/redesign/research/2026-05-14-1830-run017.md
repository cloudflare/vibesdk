## 1. Mastra — deltas vs run011

### v1.33.0 (May 13 2026) — most relevant for vibesdk

**CF Workers MCP compatibility fix:**
- `@mastra/mcp`: `jsonSchemaValidator pass-through` for MCPClient and MCPServer
- V8 isolate environments (CF Workers) block `new Function()` compilation
- This fix means Mastra MCP tooling now works in CF Workers without workarounds
- **vibesdk implication:** Mastra's MCP layer is directly usable in vibesdk's CF Worker environment. No custom MCP shims needed.

**ResponseCache (new):**
- `ResponseCache` input processor: skips model LLM calls by replaying cached responses per workflow step
- Cache key: step input signature
- **vibesdk implication:** Phase plan generation is expensive (~4k tokens prompt, ~2k tokens completion). ResponseCache could serve repeat identical-plan requests without LLM cost. Apply to `PhaseWorkflow.plan` step. Estimate: 10-30% cost reduction on repeat patterns.

**Agent Signals (new):**
- Agents accept contextual signals mid-run → "reliable follow-ups while responses stream"
- Enables injecting external context (user input, eval result) without terminating the current run
- **vibesdk implication:** Could replace the current abort-and-restart pattern for user interruption in `phasic.ts`. Agent Signals would let vibesdk inject "user changed requirement" as a signal mid-phase without full abort.

**OTEL log forwarding + MODEL_INFERENCE spans:**
- `MODEL_INFERENCE` spans for every LLM call in a workflow
- OTEL log forwarding to Datadog/Grafana
- **vibesdk implication:** PhaseWorkflow already emits STATE_DELTA; adding OTEL spans would give infra-level observability for the Mastra eval path. Zero-cost addition — just configure the OTEL exporter.

## 2. Mem0 — deltas vs run011

### 2026 State Report (published May 2026)

**New algorithm benchmark (confirmed production-grade):**
```
# schema: benchmark|score|tokens-per-query|notes
LoCoMo|91.6|6,956|+29.6 pts temporal, +23.1 pts multi-hop vs old algo
LongMemEval|93.4|6,787|strong on longitudinal recall
BEAM-1M|64.1|6,719|1M token corpus
BEAM-10M|48.6|6,914|10M token corpus (still competitive)
```

**4× token reduction:** Previous algorithm consumed ~26,000 tokens/query. New: ~7,000. At Gemini Flash pricing (~$0.075/1M input), that's a quarterly saving of ~72% on memory retrieval costs at vibesdk's scale.

**Architecture improvements:**
- Single-pass hierarchical extraction (agent-generated facts = first-class)
- Multi-signal retrieval: semantic similarity + BM25 keyword + entity matching
- Built-in entity linking → no external graph store required (was needed in 2025)
- Async-first memory writes → zero latency penalty on `remember()` calls

**@mastra/mem0 (first-party TS integration):**
- Package: `@mastra/mem0`
- No Python server, no Node native bindings
- TS-first, works inside Mastra workflows
- **vibesdk implication:** If `@mastra/mem0` is CF Workers compatible (to verify — check if it uses fetch internally), it could replace `Mem0RestMemoryClient`. The REST client we built is already fetch-based, so the underlying interface is the same. Migration would be: swap `new Mem0RestMemoryClient(key)` → `new Mastra Mem0Client(key)` inside `MemoryClient` interface. Needs one test run in workerd to confirm.

**Integration ecosystem:** 21 frameworks including Mastra, LangChain, OpenAI Agents SDK, CrewAI, Google ADK.

**Current state (v0.24.1, Apr 23 2026):**
- GitHub Action: `@letta-code` tag in issues/PRs triggers async review
- Auto-review config: `on: pull_request: types: [opened]` with review prompt
- Memory-first: persisted agent remembers repo-specific patterns across ALL PRs (cross-session memory)
- v0.24.1 change: "refactored tools to surface Task tool as Agent for Claude Code parity"

**Comparison:**
```
# schema: capability|letta-code-action|vibesdk-codeDebugger
trigger|async (PR opened, @letta-code tag)|synchronous (user requests debug)
model|model-agnostic (Claude/GPT/Gemini/GLM)|Gemini 2.5 Pro (32k reasoning)
memory|persists repo patterns across PRs|in-session only (DO state)
scope|code review, bug analysis, PR feedback|in-session fix (regenerate_file)
```

**Verdict:** These are **different tools for different workflows**. letta-code-action is async PR review with cross-session repo memory. vibesdk's codeDebugger is real-time in-session bug fixing with deep reasoning (32k token budget). Not substitutes — complementary. letta-code-action could be a FUTURE vibesdk feature (add `@vibesdk-debug` GitHub Action for repo-level review), not an immediate gap.

## What this means for vibesdk (tech pillar, cycle 4)

1. **Mastra v1.33.0 MCP compat fix is immediately actionable.** The `jsonSchemaValidator pass-through` fix means `@mastra/mcp` now works in CF Workers (V8 isolate `new Function()` restriction addressed). vibesdk's S9 MCP server plan (from run016: expose `create_phase`, `run_phase`, `get_phase_status`, `get_eval_verdict`) can use `@mastra/mcp` directly — no custom MCP shim needed. Reduces estimated MCP server effort from 2 sprints to ~1.5 sprints.

2. **Verify @mastra/mem0 CF Workers compat.** The package claims TS-first with no native bindings. If it uses fetch internally (likely, given Mem0 REST is fetch-based), it would be a drop-in replacement for `Mem0RestMemoryClient` AND adds Mastra workflow integration out of the box. One test in `wrangler dev` to confirm: `import { MastraMem0MemoryClient } from '@mastra/mem0'` → run in workerd context. Low effort, high value if confirmed.

3. **ResponseCache in PhaseWorkflow.plan is a cost win.** Repeat plan generation for identical (blueprint, query) pairs can skip the LLM call entirely. PhaseWorkflow.plan is the most expensive step (~4-6k prompt tokens). ResponseCache adds a deterministic cache layer before the LLM call. Estimate: 1-2 days implementation, 10-30% LLM cost reduction on repeat-pattern projects.

4. **Mem0 4x token reduction changes memory ROI math.** Run011 noted Memori's claim of 81.95% recall at 4.97% of full-context tokens. Mem0's new algorithm achieves comparable recall (91.6% LoCoMo) at ~7k tokens/query (was ~26k in 2025). At Gemini Flash pricing, memory retrieval is now within budget even at high frequency. Recalculate memory adoption threshold: previously "expensive" use cases are now cost-neutral.

5. **CF Agent Memory private beta status extends Mem0 as correct choice.** ADR-006 Mem0 pilot decision is validated for another cycle. No change needed — continue with `Mem0RestMemoryClient` in S9, migrate to `AgentMemoryClient` when GA + pricing announced.

6. **Zod v4 codemod existence slightly de-risks S10 dep upgrade.** The `zod-v3-to-v4` codemod handles ~70% of mechanical rewrites. Revised estimate: 6-8 days (down from 8-10) if run as an isolated worktree sprint. DEFER decision unchanged, but S10 timeline is slightly tighter.

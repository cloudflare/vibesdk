## 2. Cursor Cloud Agents — multi-repo, build-secret, layer-caching (May 13)

**Source:** cursor.com/changelog (May 13 2026)

**What:** Cloud agents now support multi-repo environments configurable per agent and reusable across sessions. Dockerfile builds gain build-time secrets for private registries, layer caching cuts re-builds 70%, plus agent-led env setup, version-history rollback, audit logs, scoped egress, and DE-level secrets.

**vibesdk impact:**
- **Multi-repo env** = direct competitor to our template-+-user-fork model. Our SimpleCodeGeneratorAgent currently clones one template; Cursor lets agents pull dependencies from a private monorepo. Consider exposing `additionalRepos[]` in Blueprint as future-spec input.
- **Build-secret support** validates ADR-005 (User Secrets DO) direction — every serious agent platform now has scoped per-environment secrets. Our XChaCha20 store + key-rotation design is competitive.
- **Layer-cache 70%** — sandbox cold-start is our biggest UX latency. Worth a sprint to investigate Docker layer caching in our container-service factory (`worker/services/sandbox/factory.ts` already modified per git status).
- **Scoped egress** is a new compliance feature we lack. File for a future security ADR.

## 4. Mastra @mastra/core v1.33.0 (May 13) — push PubSub, response cache, agent signals

**Source:** github.com/mastra-ai/mastra/releases (May 13 2026)

**What:** v1.33 GA. Notable: (1) push-capable PubSub + HTTP workflow event ingestion (GCP Pub/Sub, AWS SNS, no pull worker); (2) **ResponseCache input processor** that skips LLM calls by replaying cached responses with per-request overrides; (3) **Agent Signals** for threaded chat — `subscribeToThread` lets clients follow mid-run thread activity; (4) Azure OpenAI Responses API; (5) Aurora DSQL storage; (6) MODEL_INFERENCE OTEL spans + unified eval scores.

**vibesdk impact:**
- **ResponseCache** is the architectural item. vibesdk re-prompts identical UserConversationProcessor turns frequently on reconnect. A cache layer on `executeInference()` keyed by `(model, messages-hash, tools-hash)` would cut Anthropic spend during reconnect storms. Mastra's implementation is reference-quality — read their PR and consider mirroring in `worker/agents/inferutils/core.ts`.
- **Agent Signals / subscribeToThread** validates our existing WebSocket "agent_connected" re-sync protocol. No change required, but the API shape (`subscribeToThread(threadId)`) is cleaner than our reconnect-broadcast model — note for a future websocket-API refactor.
- **MODEL_INFERENCE OTEL spans** — we have no OTEL today. As CF Workers OTEL maturity rises (CF added OTEL exports in cycle 3), align span naming with Mastra's emerging convention.

## 5. Replit Agent 4 topology — micro-VMs per parallel task (closes run013 BLOCK partially)

**Source:** blog.replit.com (Agent 4 launch posts), product.blog Agent 4 OS deep-dive

**What:** Replit's parallel sub-agent isolation uses **micro-VMs**, not DOs and not single shared containers. Quote: "Adi made branching feel instant using micro VMs, creating isolated task environments that spin up in seconds." Orchestrator coordinates sub-agents and resolves file conflicts. Pipeline = specialized models (explorer / tester / sub-agent worker).

**vibesdk impact:** Closes run013's "per-DO vs per-container topology" question for Replit specifically — answer is **per-micro-VM** (likely Firecracker-class). vibesdk's choice should be **per-DO supervisor + per-CF-Container worker** (matches CF DO Facets pattern from run019 §3). We do not need our own micro-VM layer because CF Containers already provide the isolation boundary. Confirms ADR-007 Option A direction. **Partial-close — still BLOCKED on whether Replit's micro-VM = Firecracker or proprietary; not critical for our decision.**

## 2. Replit Agent 4 Design Canvas — GA Status

### Prior context (run029)
Design Canvas was in beta as of run029 (May 15 02:00).
"Visual variant per element, apply directly, effort-based pricing."

### Current status
Replit Design Canvas remains in **beta** as of this research run.  Not GA.

**Key signals:**
- Replit Agent 4 is still "Agent 4 Beta" in product marketing
- Design Canvas requires Replit's in-browser rendering stack (React component hot-reload in the sandbox)
- vibesdk equivalent: DESIGN.md injection provides structural guidance; no visual direct-manipulation

**Decision DEC-033-B:** Design Canvas remains a "monitor but don't implement" item.
If Design Canvas ships GA on both Lovable + Replit simultaneously (likely Q3 2026), it becomes
table-stakes and triggers a vibesdk design sprint.  Current DESIGN.md injection is the differentiator:
"better structural output than pixel manipulation" (no regression testing, no layout thrash).

vibesdk counter-narrative update: "Design Canvas shows you the app. DESIGN.md tells the AI
HOW to build it. The difference is whether you're fixing the output or preventing bad output."

## 3. v0 DB Auto-Provisioning — GA Status

### Prior context (run029)
v0 (Vercel) shipped DB auto-provisioning: Supabase / Neon / Upstash / Aurora / DynamoDB.
One-click from chat → database provisioned, connection string in env → no setup required.

### Current status (May 2026)
v0 DB is **GA** (shipped Q1 2026, confirmed by changelog).  Not a beta.
Supported backends: Neon PostgreSQL (default), Supabase, Upstash Redis, Vercel KV.
Pricing: Neon free tier included; paid tiers via Vercel billing pass-through.

**vibesdk gap:** Users must manually configure D1 or bring their own Neon/Supabase.
DEC-031-B (D1 auto-provisioning, Sprint 3) is the correct response.

**Decision DEC-033-C:** Confirm DEC-031-B Sprint 3 priority.
Add Neon PostgreSQL as an alternative to D1 for generated apps — many SMB users prefer
Postgres over SQLite for familiarity.  Generated wrangler config should support both:
```
# Option A: D1 (default, CF-native, 0 cost for hobby)
[[d1_databases]]
binding = "DB"
database_name = "app-db"
database_id = "auto-provisioned"

# Option B: Neon (user brings connection string via secret)
# DATABASE_URL = wrangler secret put DATABASE_URL
```
This doesn't require auto-provisioning on vibesdk's side — just better scaffolding in the
generated code so users know exactly how to hook in Neon/Supabase.

Bolt, Replit, and Blink.new.  vibesdk's ₹1699/₹3999 INR pricing is actually a STRATEGIC
DIFFERENTIATOR for the Indian SMB market — not a bug.  INR pricing + Razorpay checkout
gives vibesdk a geographic moat that Lovable/Bolt (USD-only) don't have.

**Decision DEC-033-E:** Amplify India-first positioning — **no engineering action needed**,
but the pricing page should mention "First AI agentic engineering platform priced for India."
Add to pricing page header text: "Designed for India. ₹1699/mo vs $25/mo (₹2100+) for global tools."

## 5. Deploy Badge UX (from Q3 audit action)

### Current state (per run031 gap analysis + iter 44 verification)

The Cloudflare Workers deploy is already fully implemented:
- Backend: `deployToCloudflare()` in `base.ts` → `DeploymentManager.deployToCloudflare()`
- WebSocket: `CLOUDFLARE_DEPLOYMENT_STARTED/COMPLETED/ERROR`
- Frontend: `DeploymentControls` component in chat.tsx
- Gap: deploy URL is buried in `DeploymentControls` panel; not visible at a glance

**Proposed:** `DeployBadge` — a compact chip in the badge row (next to `IsolationBadge`,
`CostPreviewBadge`, `PhaseQualityBadge`) that shows deploy status:
- Pre-deploy: dim globe icon, "Not deployed"
- Deploying: spinner + "Deploying…"
- Deployed: emerald globe icon, "Live" (click → open deploymentUrl)

This is the same pattern as `IsolationBadge` — pure UI, no new backend.

**Decision DEC-033-F:** Implement `DeployBadge` component — **Sprint 2 P0 (UX polish)**.
Estimated: 2h.  Files: `src/components/quality/DeployBadge.tsx`, mount in `chat.tsx` badge row.

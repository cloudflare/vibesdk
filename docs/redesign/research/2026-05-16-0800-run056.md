**Source:** cursor.com/changelog May 2026

```

Slack updates|Alert on finding|Not relevant (different workflow)
MCP support|Extensible scanning rules|Not relevant
```

### Scanning vs structural immunity
Cursor's approach = **reactive scanning** (find problems in code as it arrives). Lovable's Wiz integration = same.

vibesdk's approach = **structural prevention** (DO per-session isolation prevents the entire BOLA class; no code path exists to access another user's data).

```
# schema: approach|finds_existing_bugs|prevents_class|user_trust_message
Code scanning (Cursor/Wiz/Lovable)|YES|NO (scanner missed a bug class → breach still possible)|"We scan for problems"
DO structural isolation (vibesdk)|N/A|YES (data physically separate by architecture)|"Problems cannot exist"
```

**Positioning confirmed correct.** Existing /security page + /blog/lovable-bola already makes this argument. Cursor Security Review validates that competitors are aware of the security positioning problem and responding with scanning (not architecture). No change to vibesdk positioning needed — narrative is already more defensible.

### Update note for /security page
One sentence addition on the "scanning vs architecture" framing (optional enhancement):
> "Code scanners — including those used by Cursor, Lovable, and others — find vulnerabilities after they're written. Per-DO isolation ensures the vulnerability class doesn't exist to begin with."
